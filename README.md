# Pragmatic Games — каталог

Одностраничное приложение: каталог из ~1000 игр Pragmatic Play с ленивой
загрузкой при скролле, поиском, фильтром по типу игры и детальной карточкой игры.
Тестовое задание на позицию Frontend-разработчика.

> **Демо:** _будет добавлено после настройки деплоя на GitHub Pages (см. «Деплой»)._

---

## Возможности

- Список игр (превью + название), данные — из реального API.
- **Ленивая загрузка при скролле** на `IntersectionObserver` — без сторонних библиотек.
- **Виртуализация грида** — в DOM держится только видимая часть карточек (десятки, не тысячи), независимо от размера каталога.
- **Поиск** по названию (по кнопке, с валидацией пустого запроса) и **фильтр по `gameTypeID`** — целиком на фронте (API их не поддерживает).
- Fallback для битых превью, skeleton при загрузке, состояния loading / error / empty.
- Адаптивная тёмная вёрстка, доступность (навигация с клавиатуры, aria, focus-visible).

## Что сделано сверх ТЗ и зачем

ТЗ оценивает «строение и расширяемость архитектуры, читаемость кода, соответствие
макету». Ниже — что добавлено поверх минимума и с какой целью.

- **Детальная карточка игры** (`/game/:id`, lazy-роут) — демонстрирует, что архитектура
  готова расти за пределы одного экрана: роутинг, code-splitting и переиспользование
  кэша каталога (отдельного эндпоинта на игру в API нет). Наглядно показывает
  расширяемость, а не декларирует её.
- **Виртуализация грида** (`useWindowVirtualizer`) — ТЗ требует только ленивую загрузку,
  но на 1000+ карточках без виртуализации DOM распухает и скролл проседает. Держим в DOM
  только видимое окно.
- **FSD с машинной проверкой границ слоёв** (`import-x/no-restricted-paths`) — нарушение
  архитектуры становится ошибкой сборки, а не договорённостью на словах. Прямой ответ на
  критерий «расширяемость архитектуры».
- **Типизированный расширяемый стор** — один RTK Query API + `injectEndpoints`, `RootState`
  дополняется слайсами через declaration merging. Добавление фичи не требует правок в `shared`.
- **CI (GitHub Actions)** — production-readiness: каждый push прогоняется через
  lint → typecheck → test → build.
- **Юнит-тесты (Vitest + Testing Library)** — фиксируют ключевую логику (фильтрация, утилиты,
  рендер карточки), хотя ТЗ их не требует.
- **Доступность** — клавиатура, `aria-live`, `role="alert"`, `:focus-visible`,
  `prefers-reduced-motion`.
- **Строгий тулинг** — Stylelint, Prettier, husky + lint-staged + commitlint: единый стиль
  и conventional-коммиты в репозитории «на проверку».

## Архитектура — Feature-Sliced Design

Код разбит по слоям [FSD](https://feature-sliced.design/). Зависимости направлены
строго сверху вниз, а границы **проверяются ESLint-ом** (`import-x/no-restricted-paths`),
поэтому нарушение слоёв — это ошибка сборки, а не договорённость на словах.

```
app → pages → widgets → features → entities → shared
```

| Слой       | Ответственность                                     | Здесь                                        |
| ---------- | --------------------------------------------------- | -------------------------------------------- |
| `app`      | Инициализация, провайдеры, роутер, глобальные стили | store, RouterProvider, layout, дизайн-токены |
| `pages`    | Страницы-композиции                                 | `games`, `game-details` (lazy), `not-found`  |
| `widgets`  | Самостоятельные блоки экрана                        | `games-catalog` (грид + управление)          |
| `features` | Пользовательские сценарии                           | `game-filters` (поиск + фильтр по типу)      |
| `entities` | Бизнес-сущности                                     | `game` (API, модель, `GameCard`, утилиты)    |
| `shared`   | Переиспользуемое без бизнес-логики                  | UI-кит, хуки, стор-обвязка, конфиг, base API |

Каждый слайс отдаёт наружу только через `index.ts` (public API); глубокие импорты запрещены линтером.

## Ключевые технические решения

**Проксирование API (CORS).** Апстрим `belparyaj.com/pragmatic` не отдаёт CORS-заголовки,
поэтому браузер обращается к same-origin `/api`, а не напрямую. В dev это делает
`server.proxy` Vite — ключей и переменных окружения не требуется. Для прод-хостинга
нужен аналогичный прокси на стороне раздачи (см. «Деплой»).

**Ленивая загрузка + виртуализация — две разные роли.** `useInfiniteScroll`
(IntersectionObserver + sentinel) отвечает за то, _сколько игр «загружено»_, а
`useWindowVirtualizer` — за то, _сколько узлов реально в DOM_. Виртуализатор сам считает
число колонок от ширины контейнера (как `auto-fill minmax`), позиционирует карточки
абсолютно внутри спейсера полной высоты и throttle-ит чтения скролла через `rAF`.
Результат: на 1017 играх в DOM держатся только видимые карточки (десятки, не тысячи).

**Расширяемый типизированный стор.** Есть один экземпляр RTK Query (`shared/api`),
эндпоинты подключаются через `injectEndpoints`. Тип `RootState` пустой в `shared`, а
каждый слайс дополняет его через declaration merging — стор остаётся полностью
типизированным, но `shared` ничего не знает о конкретных фичах.

**Картинки.** `shared/ui/Image`: нативный `loading="lazy"`, фиксированный aspect-ratio
(нет CLS), skeleton во время загрузки и `onError`-fallback для несуществующих превью (404).

**Поиск и фильтр.** Поиск применяется только по сабмиту (кнопка Search / Enter), пустой
запрос отсекается валидацией, а крестик сбрасывает результаты. Фильтрация — чистая функция
`filterGames` (легко тестируется). Список типов для фильтра вычисляется из данных, а не хардкодится.

## Доступность

Семантический список игр, поиск и фильтр доступны с клавиатуры, у превью — `alt`, ошибка
валидации поиска помечена `role="alert"` и `aria-invalid`, счётчик результатов и пустое
состояние объявляются через `aria-live`, глобальный `:focus-visible`, поддержка
`prefers-reduced-motion`.

## Тесты

Vitest + Testing Library покрывают ключевую логику: фильтрацию (`filterGames`), утилиты
типов, построение URL превью и рендер `GameCard`.

## CI/CD и деплой

[GitHub Actions](./.github/workflows/ci.yml) на каждый push/PR гоняет `lint → lint:styles
→ typecheck → test → build`.

Деплой планируется на **GitHub Pages через GitHub Actions** (пока не настроен).
Важное ограничение: Pages — статический хостинг без серверных rewrite, поэтому при
настройке потребуется отдельно решить два вопроса — проксирование `/api` (у апстрима нет
CORS) через внешний прокси и раздачу SPA по субпути `/<repo>/` (`base` в Vite + fallback
для deep-link’ов).
